# Prioritize Skill

![Иллюстрация скилла](assets/illustration.png)

Скилл ранжирует список требований через RICE / ICE / MoSCoW / Kano. Сам выбирает подходящий фреймворк по данным и контексту решения. Часть фреймворка Personal Corp.

## Что это

У вас есть бэклог из 10-50 идей/задач/фич — а ресурс ограничен. Нужно решить, что делать **этим** циклом, что отложить, что вообще не делать.

Скилл берёт список, прогоняет через выбранный (или сам подобранный) фреймворк приоритизации, и выдаёт три артефакта: таблицу с числовыми оценками, квадрант Impact × Effort и предложение по распределению задач на Sprint'ы. Каждая оценка с обоснованием — можно спорить с цифрами, а не «потому что так чувствую».

## Когда применять

- **Конец квартала / планирование цикла** — есть бэклог, надо отрезать что-то от Must Have
- **Команда не согласна** — нужны прозрачные критерии, чтобы спор шёл по числам, а не по громкости голоса
- **Слишком много идей** — мозговой штурм дал 30 пунктов, остаётся 2-3 спринта
- **Перед PRD** — определить top-5, под которые имеет смысл писать детальные требования

Не для: одной задачи (приоритизировать нечего), кризисных решений «сейчас или никогда» (нет времени на оценку), стратегических развилок без данных (нужен другой инструмент).

## Что получишь на выходе

1. **Таблица оценок** — каждый пункт с числами по каждой размерности фреймворка (R/I/C/E для RICE, и т.п.) + итоговый score + однострочное обоснование
2. **Impact × Effort 2×2** — наглядно: Quick Wins / Strategic / Fill-ins / Avoid
3. **Распределение по Sprint'ам** — что в ближайший цикл, что в следующий, что в Won't Have
4. **Decision log** — почему A победил B, какие оценки спорные, что валидировать экспериментом

## Установка

```bash
cp -r skills/prioritize ~/.claude/skills/
```

После этого скилл доступен в Claude Code.

## Настройка (опционально)

Скилл работает из коробки — спросит недостающие параметры при первом запуске. Если хочешь стабильные дефолты между прогонами, добавь в `CLAUDE.md` проекта секцию `## Prioritize Config`:

| Конфиг | Назначение |
|--------|-----------|
| Default framework | RICE / ICE / MoSCoW / Kano — если хочешь один и тот же фреймворк всегда |
| Sprint capacity | Сколько person-days или Story Points в спринте — для шага распределения |
| Backlog source | Откуда брать список: `gh-issues`, `github-project`, `tasks-file`, `paste` |
| `gh_owner` / `gh_repo` / `gh_label` | Параметры GitHub-фильтра, если источник = `gh-issues` |
| `tasks_file` | Путь к локальному markdown-бэклогу |

Полный шаблон конфига — в `SKILL.md`.

## Как запустить

Просто скажи агенту, что нужно — естественным языком:

> «Приоритизируй мой бэклог. Список в `docs/backlog.md`, ресурс — 20 person-days в спринт, фокус квартала — retention.»

Или через slash-команду: `/prioritize`

Или коротко, если конфиг уже заполнен:

> «Ранжируй открытые issues с лейблом `backlog` в моём основном репо.»

Что произойдёт:
1. Если бэклога нет в реплике — скилл спросит откуда взять (paste / файл / GitHub фильтр)
2. Если фреймворк не указан — сам предложит подходящий с обоснованием (например, «у тебя есть Reach в DAU → берём RICE»). Ты подтверждаешь или меняешь
3. Прогон по фреймворку, калибровка (например, ≤ 50% пунктов могут получить Impact = 3 — иначе всё одинаково важно)
4. Три артефакта на выходе + decision log

## Ключевые правила

- Каждая оценка — с однострочным обоснованием, без «потому что так»
- Effort включает design + dev + QA + integration, не только разработку
- При Confidence < 80% — пометка «проверить экспериментом», не блокирует Must
- Cap на Must Have: ≤ 60% — иначе нет приоритизации, есть просто бэклог
- Скилл не принимает решение за команду — выдаёт прозрачную рекомендацию

## Связь с другими скиллами

- **`weekly-planning`** — берёт ранжированный бэклог отсюда и выбирает по нему недельные OKR / outcomes. Приоритизация **питает** выбор OKR, не заменяет его.
- **`weekly-retro`** — поставляет в бэклог новые задачи из ретро и переходящие пункты с прошлой недели.
