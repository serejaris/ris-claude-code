# Чеклист отсутствующих артефактов

При аудите проверять наличие каждого артефакта. Отсутствующие — включать в секцию 7 (контуры внедрения) как рекомендации.

## Формат рекомендации

```
## [Название] — ОТСУТСТВУЕТ

Симптом: [что наблюдается]
Риск: [что может пойти не так]
Минимум: [конкретный файл / ритуал / таблица]
Приоритет: Критичный / Высокий / Средний
```

## Классификация находок

| Тип разрыва | Описание | Пример |
|-------------|----------|--------|
| **Контроль** | Нет механизма предотвращения ошибки | Нет списка запретов → агент пушит в prod |
| **Видимость** | Нет инструмента наблюдения | Нет политики свежести → stale данные как актуальные |
| **Согласованность** | Нет общего понимания | Нет определений метрик → revenue считается по-разному |

## Матрица приоритизации

| | Необратимое | Обратимое |
|---|---|---|
| **Часто** | Критичный | Высокий |
| **Редко** | Высокий | Средний |

## Категория 1: Стратегия и цели

### 1.1 North Star Metric (NSM)
- **Что:** одна метрика, отражающая момент ценности для клиента
- **Симптом отсутствия:** разные люди называют разные "главные метрики"; нет консенсуса что считать успехом
- **Минимум:** одно предложение: "Наш NSM = [X], потому что когда клиент достигает [момент ценности], мы растём"
- **Приоритет:** Критичный

### 1.2 OMTM (одна метрика текущего фокуса)
- **Что:** тактический приоритет на 6-12 недель — не вечная звезда (NSM), а текущий фокус
- **Симптом:** одновременная работа над удержанием, конверсией и awareness без явного приоритета
- **Минимум:** statement: "OMTM этого цикла — [метрика], цель — [X], дедлайн — [дата]"
- **Приоритет:** Высокий

### 1.3 Quarterly OKR (минимальный)
- **Что:** 1-3 цели + 2-4 измеримых результата на каждую, цикл — квартал
- **Симптом:** нет ответа на "что изменится через 90 дней?"; бэклог растёт без фильтра
- **Минимум:** один файл: 1 Objective + 3 Key Results + еженедельный check-in (on-track / at-risk / done)
- **Приоритет:** Высокий

### 1.4 Weekly check-in ритуал
- **Что:** 15-минутный еженедельный ритуал: что двинулось к OKR, что заблокировано, что убираем
- **Симптом:** OKR написаны, но к ним не возвращаются до конца квартала
- **Минимум:** шаблон в ретро: "OKR status / OMTM delta / блокеры / что убираю". 5 строк
- **Приоритет:** Высокий

## Категория 2: AI-native артефакты

### 2.1 Контекст для агента (CLAUDE.md / AGENTS.md)
- **Что:** машиночитаемый файл с границами, каноническими путями, routing
- **Симптом:** агент использует устаревшие цены; путает репо; нарушает naming; делает необратимое без подтверждения
- **Минимум:** один файл: boundaries (что нельзя), canonical files, task routing. 50-100 строк
- **Приоритет:** Критичный

### 2.2 Библиотека промптов с версиями
- **Что:** промпты по задачам (анализ, рассылка, SEO) с версиями в git
- **Симптом:** "раньше работало лучше, но я изменил промпт"; один сценарий решается каждый раз заново
- **Минимум:** папка `prompts/` или `.claude/skills/`, файлы по задачам, git = история
- **Приоритет:** Высокий

### 2.3 Runbook для агентных пайплайнов
- **Что:** для критических пайплайнов: как запустить, остановить, откатить, кому эскалировать
- **Симптом:** при сбое контекст восстанавливается по памяти; каждый сбой = форс-мажор
- **Минимум:** для каждого pipeline: trigger → steps → failure modes → rollback → escalate-to
- **Приоритет:** Высокий

### 2.4 Список запрещённых действий
- **Что:** явный список действий, которые агент не делает без подтверждения
- **Симптом:** агент пушит в main; меняет цены; создаёт issues с личными данными в публичном репо
- **Минимум:** секция `🚫 Never` в AGENTS.md. 5-10 строк. Обновляется после инцидентов
- **Приоритет:** Критичный

### 2.5 Eval-набор (golden test cases)
- **Что:** тестовые примеры с известными правильными выходами для критических задач
- **Симптом:** обновили промпт → "кажется работает" → никто не проверял
- **Минимум:** 3-5 примеров ввод/ожидаемый-вывод на каждый критический пайплайн
- **Приоритет:** Средний

## Категория 3: Инфраструктура данных

### 3.1 Единый источник правды (SSOT per domain)
- **Что:** явное назначение: "X владеет цифрами Y, не брать из других мест"
- **Симптом:** два документа называют разное число пользователей/клиентов; агент берёт цену из устаревшего файла
- **Минимум:** таблица: метрика / файл-владелец / дата обновления. 10-15 строк
- **Приоритет:** Критичный

### 3.2 Определения метрик (глоссарий)
- **Что:** для каждой метрики — формула, что включается, что исключается
- **Симптом:** "выручка $X" — с pending или без? Включает допродажи/подписку? Никто не может ответить
- **Минимум:** раздел: метрика = формула + включено + исключено + единица
- **Приоритет:** Критичный

### 3.3 Политика свежести данных
- **Что:** SLA на каждый data source + правило что делать если данные устарели
- **Симптом:** snapshot 3-недельной давности используется как current state без дисклеймера
- **Минимум:** в SSOT-таблице колонка `freshness_sla` (daily / weekly / per-cycle) + правило дисклеймера
- **Приоритет:** Высокий

### 3.4 Протокол исторических снимков
- **Что:** при изменении канонического числа — сохранять предыдущее с датой
- **Симптом:** невозможно ответить "сколько было пользователей/клиентов в цикле N?"
- **Минимум:** секция `## History` с датированными снимками ключевых метрик
- **Приоритет:** Высокий

## Категория 4: Governance

### 4.1 Журнал решений (Decision Log)
- **Что:** дата / вопрос / решение / причина / отвергнутые альтернативы
- **Симптом:** одни и те же вопросы (цена? день недели? формат?) обсуждаются заново каждый цикл
- **Минимум:** один файл, 3-4 строки на запись. Не нужен ADR-формат
- **Приоритет:** Высокий

### 4.2 Матрица ответственности (упрощённый RACI)
- **Что:** domain / owner / кто консультируется. Для 1-5 человек достаточно Owner + Stakeholder
- **Симптом:** цены меняются в разных местах разными людьми; нет одного ответственного за canonical data
- **Минимум:** таблица 5-10 строк: domain / single owner / review required from
- **Приоритет:** Высокий

### 4.3 Правила эскалации
- **Что:** пороги: при каком условии агент / куратор должен остановиться и спросить
- **Симптом:** либо всё эскалируется (перегрузка), либо ничего (необратимые ошибки)
- **Минимум:** 5-7 правил формата "если [условие] → [стоп/спроси/действуй]"
- **Приоритет:** Критичный

### 4.4 Журнал инцидентов
- **Что:** сбои и нарушения: что произошло, причина, исправление, как предотвратить
- **Симптом:** правила обновляются хаотично после инцидентов; ошибки повторяются
- **Минимум:** секция или файл: дата / что случилось / root cause / fix / правило добавлено
- **Приоритет:** Средний
